## Architecture

Next.js App Router site for a local plumbing business ("The Plumbing Stars"). All routes and components live under `src/` (`src/app/`, `src/components/`, `src/lib/`) — there is no parallel top-level `components/` directory.

**Next.js 16 + React 19** — both are cutting-edge; see AGENTS.md before writing any framework code.

**Route map:**
- `/` — main landing page: hero, services grid, special offer, why-us, contact CTA
- `/services` — index of all service pages; nav prefers city-scoped URLs when a location is detected
- `/services/[slug]` — county-wide service pages (fallback when no city is chosen), built with `ServiceShell`
- `/losangeles/[city]` and `/ventura/[city]` — city hub pages (`CityHubShell`) with surrounding-community SEO
- `/losangeles/[city]/[service]` and `/ventura/[city]/[service]` — city-scoped service pages (`CityServiceShell`)
- `/losangeles` — LA county landing with city list, Leaflet map, and `CountyServiceExplorer`. Target of `theplumbingstarslosangeles.com` — keep top-level
- `/ventura` — Ventura county landing (same shell). Target of `theplumbingstarsventura.com` — keep top-level
- `/search` — search by city or ZIP (noindexed)
- `/schedule` — booking page wrapping `BookingForm` (noindexed)
- `/contact` — contact form page
- `/coupons`, `/faq`, `/reviews`, `/leave-review` — content/marketing pages
- `/api/book`, `/api/contact`, `/api/review` — form POST endpoints (Resend / Gmail)
- `/sitemap.xml`, `/robots.txt` — generated by `src/app/sitemap.ts` / `src/app/robots.ts`

Old/retired URLs (`/service-areas`, `/service-areas/los-angeles-county`, `/service-areas/ventura-county`, `/areas`, `/resources`, retired city hubs) permanently redirect (308) via `next.config.ts`.

**Component roles** (all in `src/components/`):
- `Header` — sticky nav; service links via `ServicesNavLinks` (city-aware)
- `Footer` — static; links to both county pages and starred city hubs
- `MobileMenu` — `"use client"` hamburger drawer, rendered by `Header`, hidden on `md:` and up
- `Breadcrumbs` — visible trail + matching `BreadcrumbList` JSON-LD
- `JsonLd` — thin `<script type="application/ld+json">` renderer
- `BookingForm` — `"use client"` form that POSTs to `/api/book`; optional `region` prop (defaults to `"losangeles"`)
- `ServiceShell` / `CityHubShell` / `CityServiceShell` — shells for county services, city hubs, and city×service pages
- `CountyPageShell` / `CountyServiceExplorer` — county landing: ZIP search, city list, map
- `ServiceMap` / `ServiceMapClient` — Leaflet map (dynamic, `ssr: false`); polygons hardcoded in client; GeoJSON used for point-in-polygon in `areaDetection.ts`

**Data / lib:**
- `src/lib/site.ts` — shared site constants
- `src/lib/services.ts` — canonical 7 services + `servicePath` / `servicePathForLocation`
- `src/lib/cityHubs.ts` — starred hubs, retired aliases, SEO helpers
- `src/lib/surroundingCities.ts` — nearby communities per hub for copy + search routing
- `src/lib/structuredData.ts` — JSON-LD builders
- `src/lib/zipLookup.ts` — ZIP → region (+ optional centroids for retired hubs)
- `src/lib/serviceLocations.ts` — `CITY_REGION` search map + `countyPathForRegion` → `/losangeles` | `/ventura`
- `src/lib/mailer.ts` — Resend (`RESEND_API_KEY` + required `EMAIL_FROM`) with optional Gmail fallback. Non-production requires `BOOKING_TO` / `REVIEW_TO`. See `.env.example`.

**Adding a new service page:** create `src/app/services/<slug>/page.tsx`, export `metadata` (with `alternates: { canonical: "/services/<slug>" }`), add the service to `src/lib/services.ts`, define FAQs, and return `<ServiceShell …>`.

**Adding a new city hub:** add an entry to `CITY_HUBS` in `cityHubs.ts`, list surrounding communities in `surroundingCities.ts`, and keep ZIPs/`CITY_REGION` in sync. City and city×service routes are generated automatically.

**Adding a new service area county:** add ZIPs to `zipLookup.ts`, update `countyPathForRegion`, create `src/app/<region>/page.tsx` as a top-level route, and add redirects in `next.config.ts` if replacing an old URL.

## SEO conventions

- Every route sets `alternates: { canonical: "<path>" }` in its `metadata` export (relative paths resolve against `metadataBase` in the root layout).
- The root layout defines a title template (`"%s | The Plumbing Stars"`); page `title` values should be the unique part only, not the full "... | The Plumbing Stars" string.
- Pure utility/form pages with no unique indexable content (`/search`, `/schedule`, `/leave-review`) are set to `robots: { index: false, follow: true }`.
- Structured data lives in `src/lib/structuredData.ts` and is rendered via the `JsonLd` component — add new schema builders there rather than inlining `<script>` tags.
- `src/app/sitemap.ts` must be updated when routes are added or removed.

## Styling

**Tailwind CSS v4** is in use — there is no `tailwind.config.js`. Configuration is done entirely in CSS via `@theme` blocks in `globals.css`. Do not create a `tailwind.config.js`.

Brand tokens available as Tailwind utility classes:

| Token | Value |
|---|---|
| `bg-brand-navy` | `#0b2a55` |
| `bg-brand-navy-dark` | `#081d3d` |
| `bg-brand-navy-darker` | `#050f28` |
| `bg-brand-red` | `#b81f2a` |
| `bg-brand-red-dark` | `#7c1018` |
| `bg-brand-sky` | `#92b7e2` |
| `bg-brand-sky-light` | `#e9f0f9` |
| `bg-brand-gold` | `#f5c518` |

`brand-gold` is reserved for star-rating icons, not general accents. Use these tokens instead of hardcoded hex values.

Fonts: `font-sans` → Manrope (body); `font-display` → DM Serif Display (headings). `h1`–`h4` default to the display font via a global CSS rule in `globals.css`.
