## Architecture

Next.js App Router site for a local plumbing business ("The Plumbing Stars"). All routes and components live under `src/` (`src/app/`, `src/components/`, `src/lib/`) — there is no parallel top-level `components/` directory.

**Next.js 16 + React 19** — both are cutting-edge; see AGENTS.md before writing any framework code.

**Route map:**
- `/` (`page.tsx`) — main landing page: hero, services grid, special offer, why-us, contact CTA
- `/services` — index of all service pages
- `/services/[drain-cleaning|sewer-repair|hydro-jetting|camera-inspection|trenchless-replacement|pipe-lining|water-heater]` — 7 individual service pages, all built with `ServiceShell`
- `/service-areas` — hub page: county picker + ZIP lookup (`ChooseAreaClient`), links out to `/losangeles` and `/ventura`
- `/losangeles` — LA county landing with city list, Leaflet map, and `CountyServiceExplorer`. This is also the target of an external domain redirect (`theplumbingstarslosangeles.com`), so keep it a top-level path — do not nest it under `/service-areas/...`
- `/ventura` — Ventura county landing, same shell as LA. Also the target of an external domain redirect (`theplumbingstarsventura.com`) — keep it top-level
- `/search` — search by city or ZIP (noindexed utility page)
- `/schedule` — booking page wrapping `BookingForm` (noindexed utility page)
- `/coupons`, `/faq`, `/reviews`, `/leave-review` — content/marketing pages
- `/api/book` — POST endpoint that receives booking form submissions and emails them via nodemailer
- `/api/review` — POST endpoint that receives review submissions
- `/sitemap.xml`, `/robots.txt` — generated by `src/app/sitemap.ts` / `src/app/robots.ts`

Old/retired URLs (`/service-areas/los-angeles-county`, `/service-areas/ventura-county`, `/areas`, `/contact`, `/resources`) permanently redirect (308) to their current locations via `next.config.ts`. Update that file if you rename or remove a route.

**Component roles** (all in `src/components/`):
- `Header` — sticky nav with Areas / Services / Contact / Resources dropdowns; no external props
- `Footer` — static; links to both service-area pages
- `MobileMenu` — `"use client"` hamburger drawer, rendered by `Header`, hidden on `md:` and up
- `Breadcrumbs` — renders a visible breadcrumb trail *and* the matching `BreadcrumbList` JSON-LD; use on every page below a hub (services/service-areas subpages)
- `JsonLd` — thin `<script type="application/ld+json">` renderer; pass a schema object from `src/lib/structuredData.ts`
- `ChooseAreaClient` (in `src/app/service-areas/`) — `"use client"` ZIP lookup that pushes to the matching county page
- `BookingForm` — `"use client"` form that POSTs to `/api/book`; `region` prop is optional (defaults to `"losangeles"`) and is also user-selectable via a dropdown
- `ServiceShell` — server component wrapper for all service pages: renders breadcrumbs, `Service`/`FAQPage` JSON-LD, `Header`, hero, trust bar, `children` (page-specific content), and FAQ section. Requires `serviceName` and `slug` props (short label + URL slug) in addition to `title`/`intro`/`scheduleLabel`/`faqs`
- `CountyPageShell` / `CountyServiceExplorer` — county landing page logic: geolocation, ZIP search, city list, map
- `ServiceMap` — thin server-safe wrapper that uses `next/dynamic` with `ssr: false` to lazy-load `ServiceMapClient` (Leaflet requires the DOM). Accepts `county?: "la" | "ventura" | "both"` (defaults to `"both"`)
- `ServiceMapClient` — `"use client"` Leaflet map rendering LA and/or Ventura county polygons via `react-leaflet`. Polygon coordinates are hardcoded in this file

**Data / lib:**
- `src/lib/site.ts` — shared site constants (`SITE_URL`, `SITE_NAME`, `PHONE_DISPLAY`, `PHONE_TEL`, etc.) — use these instead of hardcoding the phone number or domain
- `src/lib/services.ts` — canonical list of the 7 bookable services (slug, name, description); shared by the home grid, `/services` index, and the `BookingForm` dropdown
- `src/lib/structuredData.ts` — JSON-LD builders (`localBusinessSchema`, `serviceSchema`, `breadcrumbSchema`, `faqPageSchema`, `reviewsSchema`) — pair with the `JsonLd` component
- `src/lib/zipLookup.ts` — exports `getRegionByZip(zip)` returning `"losangeles" | "ventura" | null`; backed by two hardcoded `Set`s. Add ZIPs here to expand coverage
- `src/lib/serviceLocations.ts` — exports `countyPathForRegion(region)` returning the `/service-areas/...` URL for a region; update this if county URLs ever change
- `src/lib/mailer.ts` — nodemailer via Gmail. Requires `GMAIL_USER` and `GMAIL_APP_PASSWORD` env vars. `BOOKING_TO` overrides the destination address (defaults to `info@theplumbingstars.com`). When env vars are absent the API route logs the lead to console instead of silently failing

**Adding a new service page:** create `src/app/services/<slug>/page.tsx`, export `metadata` (with `alternates: { canonical: "/services/<slug>" }`), add the service to `src/lib/services.ts`, define a `faqs` array, and return `<ServiceShell serviceName="..." slug="<slug>" title="..." intro="..." scheduleLabel="..." faqs={faqs}>` with page-specific `<section>` blocks as children.

**Adding a new service area county:** add ZIPs to `zipLookup.ts`, update `countyPathForRegion` in `serviceLocations.ts`, create `src/app/<region>/page.tsx` as a top-level route mirroring the LA/Ventura pattern (including `Breadcrumbs`), add it to `src/app/service-areas/page.tsx`'s county list, and add a redirect in `next.config.ts` if replacing an old URL.

## SEO conventions

- Every route sets `alternates: { canonical: "<path>" }` in its `metadata` export (relative paths resolve against `metadataBase` in the root layout).
- The root layout defines a title template (`"%s | The Plumbing Stars"`); page `title` values should be the unique part only, not the full "... | The Plumbing Stars" string.
- Pure utility/form pages with no unique indexable content (`/search`, `/schedule`, `/leave-review`) are set to `robots: { index: false, follow: true }`.
- Structured data lives in `src/lib/structuredData.ts` and is rendered via the `JsonLd` component — add new schema builders there rather than inlining `<script>` tags.
- `src/app/sitemap.ts` must be updated when routes are added or removed.

## Styling

**Tailwind CSS v4** is in use — there is no `tailwind.config.js`. Configuration is done entirely in CSS via `@theme` blocks in `globals.css`. Do not create a `tailwind.config.js`.

Brand tokens available as Tailwind utility classes:

| Token | Value |
|---|---|
| `bg-brand-navy` | `#0b2a55` |
| `bg-brand-navy-dark` | `#081d3d` |
| `bg-brand-navy-darker` | `#050f28` |
| `bg-brand-red` | `#b81f2a` |
| `bg-brand-red-dark` | `#7c1018` |
| `bg-brand-sky` | `#92b7e2` |
| `bg-brand-sky-light` | `#e9f0f9` |
| `bg-brand-gold` | `#f5c518` |

`brand-gold` is reserved for star-rating icons, not general accents. Use these tokens instead of hardcoded hex values.

Fonts: `font-sans` → Manrope (body); `font-display` → DM Serif Display (headings). `h1`–`h4` default to the display font via a global CSS rule in `globals.css`.
